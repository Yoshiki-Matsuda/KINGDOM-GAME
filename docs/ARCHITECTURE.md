# キングダム戦略ゲーム — 全体アーキテクチャ

## 0. 目標モード: PvPvE

**最終形は PvPvE（Player vs Player vs Environment）** を前提に設計する。

- **PvP**: 複数プレイヤーが同じワールドで領地・資源を争う。
- **PvE**: 中立勢力・AI勢力が存在し、プレイヤーはそれらとも戦闘・外交する。
- **混在**: 1つのマッチ／ワールド内で「人間同士」と「人間 vs 環境」が同時に成立する。

### 設計への反映

| 観点 | 方針 |
|------|------|
| 領地・所有者 | `owner_id` は「プレイヤーID」または「AI/中立勢力ID」を区別できる形にする。 |
| 認証・マッチング | 将来、プレイヤー識別・ルーム/ワールド単位の参加者管理が必要。 |
| ターン・同期 | セミリアルタイム／ターン制のまま、全参加者（人間＋AI）が同一ルールで行動。 |
| フェーズ | 戦闘解決・ターン終了の順序を、PvP でも PvE でも一貫して扱えるようにする。 |

実装はまず **PvE のみの最小MVP** から始め、その上にプレイヤー複数・マッチング・AI を乗せて PvPvE に拡張する。

### マスレベル・地形・戦闘の流れ

- **マスのレベル**: 各マスはレベル 1〜5 を持つ。レベルとマスのイラスト（地形）は 1:1 でリンクする。
  - 例: レベル 1＝平原、レベル 2＝丘陵、レベル 3＝森、レベル 4＝山地、レベル 5＝山岳。
- **PvE 難易度**: レベルが高いマスほど PvE 部分の敵が強い。戦闘解決はすべてバックエンドで行い、ユーザーはログで結果のみ閲覧する。
- **PvP は PvE の後**: 攻撃目標マスに**敵プレイヤーが援軍としてユニットを送っている**場合は、バックエンドでまず PvE（環境敵）を処理し、そのあと同じマスで **PvP**（当該援軍との戦闘）を処理する。ユーザーが見るのはログのみ。

## 1. なぜRustバックエンドがこのゲームに向いているか

### 1.1 ゲームサーバーに求められる性質

| 要件 | Rustが適している理由 |
|------|----------------------|
| **サーバー権威型** | 状態を一箇所で保持し、クライアント入力を検証する必要がある。Rustの所有権・型システムにより「状態の不正な書き換え」をコンパイル時に防ぎやすい。 |
| **イベント駆動・純粋関数** | 戦闘解決・ターン処理を「入力→出力」の純粋関数にすると、テスト・リプレイ・将来のWASM化が容易。Rustは参照透過な設計を強制しやすい。 |
| **同時接続・低レイテンシ** | WebSocketで多数クライアントと常時接続。Rustの非同期ランタイム（tokio）とゼロコピー・メモリ安全は、C++並の性能とクラッシュの少なさを両立する。 |
| **長期運用・デプロイ** | 単一バイナリでデプロイ可能。GCポーズがなく、メモリ使用量が予測しやすい。本番障害の切り分けがしやすい。 |
| **データ駆動・JSON** | `serde` によるJSON直列化が標準的。フロントと「同じ構造のデータモデル」をRustとTypeScriptで共有しやすい。 |

### 1.2 このプロジェクトでRustに期待すること

- **ゲームロジックの単一実装**: 戦闘・ターン進行はRustのみ。クライアントは表示・入力専用。
- **テスト容易性**: 純粋関数＋データ駆動で、状態遷移のユニットテストを網羅しやすい。
- **将来のWASM化の布石**: ロジックを「入力JSON → 出力JSON」に閉じれば、同じロジックをWASMに移植してクライアント側のプレビューやリプレイに使える余地を残す。

### 2.1 役割の整理

| レイヤー | 役割 |
|----------|------|
| **フロント** | 表示・入力のみ。サーバーから受け取ったJSON状態をそのまま描画し、ユーザー操作をメッセージとしてサーバーに送る。 |
| **Rustサーバー** | 権威あるゲーム状態の保持、ターン処理・戦闘の計算、WebSocketでの状態配信と行動受付。 |
| **PostgreSQL** | ゲーム・ユーザー・領地などの永続化。ターンごとのスナップショットや履歴も保存可能。 |
| **Redis** | ターン進行のロック（同一ワールドで同時に1ターンだけ進行）、セッション・プレイヤー接続情報など。 |

### 2.2 データの流れ（最小イメージ）

1. クライアントが WebSocket で接続 → サーバーが現在のゲーム状態（JSON）を送信。
2. プレイヤーが「攻撃」などを選択 → クライアントが行動メッセージを WebSocket で送信。
3. サーバーが行動を検証し、ゲームロジック（純粋関数）で新状態を計算。
4. サーバーが更新後の状態を全接続クライアントに配信。
5. 必要に応じて PostgreSQL に永続化。ターン切り替え時は Redis でロック。

## ローカルで動かす

1. **バックエンド**（ターミナル1）
   ```bash
   cd server && cargo run
   ```
   → `http://127.0.0.1:3000` で API / WebSocket が有効になる。

2. **フロントエンド**（ターミナル2）
   ```bash
   npm run dev
   ```
   → ブラウザで表示される URL（例: http://localhost:5173）を開く。

3. フロントは起動時に `ws://127.0.0.1:3000/ws` に接続し、ターン・領地一覧・地図を表示。増援・攻撃・ターン終了でサーバーに行動を送り、状態が更新される。
