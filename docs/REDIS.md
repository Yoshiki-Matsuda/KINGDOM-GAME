# Redis 利用方針（ターンロック・セッション）

## 目的

- **ターンロック**: 同一ワールド（ゲーム）で、ターン進行や戦闘解決を「同時に1処理だけ」に制限する。
- **セッション・接続管理**: プレイヤー識別やルーム参加状態の保持（PvPvE で複数プレイヤーが同一ゲームに参加する場合）。

## ターンロックのイメージ

- キー例: `game:{game_id}:turn_lock`
- 処理の流れ:
  1. 行動受信時、`SET key 1 NX EX 10` でロック取得を試行。
  2. 取得できたら `apply_action` で状態更新し、ブロードキャスト後にロック解放（DEL）。
  3. 取得できなければ「処理中」としてクライアントにエラーまたはリトライを返す。

複数ワールドがある場合は `game_id` ごとにロックするため、ワールド間は並列で問題ない。

## 実装タイミング

- 現状は **メモリ上で1ゲームのみ** のため、Redis は未導入でよい。
- **複数ゲーム（ルーム）** や **ターン制の厳密な順序** を実装する段階で、`redis` crate を追加し上記ロックを入れる。

## 接続情報（将来用）

- 環境変数例: `REDIS_URL=redis://127.0.0.1:6379`
- ローカル開発では Docker などで `redis:7` を起動して利用する想定。
